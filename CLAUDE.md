# Repository Guidelines

## Project Overview

- This is an STM32CubeMX-generated STM32H743 firmware project for Keil MDK-ARM. The main target file is `MDK-ARM/h7rec.uvprojx`.
- The project is not currently a git repository. Be extra careful to avoid broad rewrites or generated-file churn.
- Application startup is very thin: `Core/Src/main.c` initializes MPU, HAL, GPIO, RTC, SPI2, TIM2, then starts CMSIS-RTOS2 via `MX_FREERTOS_Init()`.
- `Core/Src/freertos.c` currently creates only `defaultTask`, and that task only calls `osDelay(1)` in a loop.

## Layout

- `Core/Inc` and `Core/Src`: CubeMX-generated application, peripheral, interrupt, and RTOS glue code.
- `Drivers/`: STM32H7 HAL and CMSIS device headers/sources.
- `Middlewares/Third_Party/FreeRTOS`: FreeRTOS kernel and CMSIS-RTOS2 wrapper.
- `MDK-ARM/`: Keil uVision project, startup assembly, scatter file, build outputs under `MDK-ARM/h7rec/`.
- `BSP/ST7735`: ST7735 LCD driver/BSP files. These files exist in the tree, but are not currently listed in `MDK-ARM/h7rec.uvprojx`.
- `moveset.bat`: copies `MDK-ARM/.vscode` into the root `.vscode` folder.

## Build And Tooling

- Primary build path is Keil MDK/uVision using `MDK-ARM/h7rec.uvprojx`.
- The target emits a hex file (`CreateHexFile` is enabled) into `MDK-ARM/h7rec/`.
- The active target defines are `USE_PWR_LDO_SUPPLY`, `USE_HAL_DRIVER`, and `STM32H743xx`.
- VS Code IntelliSense settings are duplicated in root `.vscode/c_cpp_properties.json` and `MDK-ARM/.vscode/c_cpp_properties.json`. The root copy appears to be generated by `moveset.bat`; avoid hand-editing only one copy unless that divergence is intentional.
- Generated build products and Keil user/runtime files are ignored by `.gitignore`; keep source changes out of `MDK-ARM/h7rec/`.

## Code Generation Rules

- Most files under `Core/` are CubeMX-generated. Preserve `/* USER CODE BEGIN ... */` and `/* USER CODE END ... */` regions, and place manual code inside user sections where possible.
- When changing peripheral configuration, prefer updating the `.ioc`/CubeMX source of truth and regenerating, then review the generated diff carefully.
- Do not casually reformat generated files; small behavioral edits are much easier to review when CubeMX formatting remains intact.

## RTOS Notes

- The project uses CMSIS-RTOS2 APIs (`cmsis_os.h`, `osThreadNew`, `osDelay`) on top of FreeRTOS.
- FreeRTOS is configured for `heap_4.c`, `configTOTAL_HEAP_SIZE` is 15360 bytes, and the default task stack is `128 * 4` bytes.
- If adding real work to the default task or adding new threads, revisit stack sizes and heap usage instead of assuming the CubeMX defaults are enough.
- HAL tick uses TIM17 through `Core/Src/stm32h7xx_hal_timebase_tim.c`; do not assume SysTick owns the HAL time base.

## LCD/ST7735 Notes

- Hardware pins for the LCD are already generated in `Core/Inc/main.h` and `Core/Src/gpio.c`: LCD DC/RS on PD13, reset on PD12, CS on PB12, backlight on PA5/TIM2_CH1, key input on PB1.
- SPI2 is configured in `Core/Src/spi.c` as master, 1-line transmit, 8-bit, CPOL low, CPHA first edge, software NSS, prescaler 16, using PB13 SCK and PB15 MOSI.
- TIM2 PWM in `Core/Src/tim.c` drives backlight brightness with prescaler `480-1`, period `1000-1`, channel 1, low polarity.
- `BSP/ST7735/lcd.c` expects one of `TFT96` or `TFT18` to be defined, but neither is currently in the Keil target defines.
- Before calling `LCD_Test()` or other LCD BSP functions from application code, add the BSP source files and include paths to the Keil project, choose the correct `TFT96`/`TFT18` define, and ensure TIM2 PWM is started before setting brightness.

## Editing Expectations

- Keep changes scoped and compatible with ARMCC/Keil unless the toolchain is intentionally migrated.
- Use HAL/CMSIS-RTOS2 APIs that match the existing project style.
- Treat `BSP/ST7735` as externally sourced/ported code: fix local integration issues narrowly, and avoid large cosmetic rewrites.
- When touching generated code and hand-written BSP code in the same change, separate the rationale clearly in the final response.
